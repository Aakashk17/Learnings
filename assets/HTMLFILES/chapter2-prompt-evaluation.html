What Is Prompt Evaluation?

Imagine you write a prompt asking Claude to summarize news articles. It works great on the three articles you tested. But does it work well on 500 different articles? Does it fail on sports news? Does it crash when the article is very long? You don't know — until you run an eval.

A prompt evaluation (or "eval" for short) is a system that tests your prompt on many different inputs, then scores how well it did. It's how professionals make sure their AI-powered apps are reliable.

📖 What the Course Teaches
The Basic Eval Idea

The course introduces evals as a repeatable testing loop: give Claude the same types of questions, check its answers, get a score. Here's a minimal eval from scratch:

basic_eval.py Python 
import anthropic

# Set up the Claude API client
client = anthropic.Anthropic()

# ── STEP 1: Define test cases ──────────────────────────────
# Each test case has an input (what we send to Claude)
# and an expected output (what a good answer looks like)
test_cases = [
    {
        "input": "What is 2 + 2?",
        "expected": "4"
    },
    {
        "input": "What is the capital of France?",
        "expected": "Paris"
    },
    {
        "input": "Who wrote Romeo and Juliet?",
        "expected": "Shakespeare"
    },
]

# ── STEP 2: Run each test through Claude ───────────────────
results = []

for test in test_cases:
    # Send the question to Claude
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=100,
        messages=[{"role": "user", "content": test["input"]}]
    )

    # Get the text from Claude's answer
    claude_answer = response.content[0].text

    # ── STEP 3: Check if the answer is correct ─────────────
    # A simple check: does the expected word appear in the answer?
    is_correct = test["expected"].lower() in claude_answer.lower()

    results.append({
        "question": test["input"],
        "got": claude_answer,
        "correct": is_correct
    })

# ── STEP 4: Print the score ────────────────────────────────
score = sum(1 for r in results if r["correct"]) / len(results)
print(f"Score: {score:.0%} ({sum(1 for r in results if r['correct'])}/{len(results)})")

This is the most basic eval: ask questions, check if the answer contains the right word, print a score.

⬆ Even Better Approach
⬆ Why upgrade? Exact string matching is fragile — "Four" fails even though it's correct.
Normalise Answers Before Comparing

Real answers come in many forms: "Paris, France", "It's Paris!", "paris". A smarter eval normalises (cleans up) text before comparing.

better_eval.py Python 
import re

def normalize(text: str) -> str:
    """
    Clean up a string so we can compare it fairly.
    'Paris, France!' becomes 'paris france'
    'FOUR'          becomes 'four'
    """
    text = text.lower()              # lowercase everything
    text = re.sub(r'[^\w\s]', '', text)  # remove punctuation
    text = text.strip()              # remove spaces at edges
    return text

def check_answer(expected: str, actual: str) -> bool:
    """
    Returns True if the expected answer appears inside
    the actual answer, after normalising both.
    """
    return normalize(expected) in normalize(actual)

# Now our grading is much more flexible:
print(check_answer("Paris", "It's Paris, France!"))  # → True ✓
print(check_answer("4", "The answer is 4."))          # → True ✓
print(check_answer("Shakespeare", "William Shakespeare"))  # → True ✓
💡 Analogy
It's Like a Spelling Bee Practice Sheet

You know the correct spellings. You give a student a test. You mark how many they got right. An eval does the exact same thing — but the "student" is your AI prompt. The higher the score, the more reliable your prompt is.

A Typical Eval Workflow

The course shows that every eval — no matter how simple or complex — follows the same four steps. Think of it as a loop you run every time you change your prompt.

1
Write (or improve) your prompt

This is the system prompt or user message template you want to test. Example: "Summarise this article in 3 bullet points."

2
Build a test dataset

A list of real inputs (and expected good outputs) that your prompt should handle well.

3
Run Claude on every test case

Send each input through your prompt and collect all the responses.

4
Grade the responses & get a score

Check each response against the expected output. Calculate a percentage score.

📖 What the Course Teaches
A Simple Eval Loop
eval_workflow.py Python 
def run_eval(prompt_template: str, test_cases: list) -> dict:
    """
    The core eval loop the course teaches.
    
    prompt_template: A string like "Summarise this: {text}"
    test_cases: A list of dicts with 'input' and 'expected' keys
    """
    client = anthropic.Anthropic()
    results = []

    for case in test_cases:
        # Fill in the template with this test's input
        full_prompt = prompt_template.format(**case["input"])

        # Ask Claude
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=500,
            messages=[{"role": "user", "content": full_prompt}]
        )

        answer = response.content[0].text

        # Simple pass/fail check
        passed = case["expected"].lower() in answer.lower()
        results.append({"passed": passed, "answer": answer})

    # Calculate final score
    num_passed = sum(1 for r in results if r["passed"])
    score = num_passed / len(results)

    return {
        "score": score,
        "passed": num_passed,
        "total": len(results),
        "results": results
    }

Add Error Handling & Progress Tracking

from tqdm import tqdm # pip install tqdm — shows a progress bar import time def run_eval_robust(prompt_template, test_cases, retries=2): client = anthropic.Anthropic() results = [] # tqdm wraps the loop and shows a live progress bar for case in tqdm(test_cases, desc="Running eval"): # Try up to (retries) times if something goes wrong for attempt in range(retries + 1): try: full_prompt = prompt_template.format(**case["input"]) response = client.messages.create( model="claude-opus-4-5", max_tokens=500, messages=[{"role": "user", "content": full_prompt}] ) answer = response.content[0].text passed = case["expected"].lower() in answer.lower() results.append({ "passed": passed, "answer": answer, "error": None }) break # success — stop retrying except Exception as e: if attempt == retries: # All retries used — record as an error, don't crash results.append({"passed": False, "answer": "", "error": str(e)}) else: time.sleep(1) # wait 1 second, then retry num_passed = sum(1 for r in results if r["passed"]) return {"score": num_passed / len(results), "results": results}
💡 Analogy
It's Like a Factory Quality Check

A car factory doesn't just build one car and assume all cars are perfect. They test every batch. They track pass rates over time. If the rate drops, something's wrong. Your eval workflow is the quality check for your AI factory.

✅ Tip
Run your eval every time you change your prompt. Even a tiny wording change can drop your score by 10%.

Generating Test Datasets

Here's the problem: writing 200 test cases by hand is exhausting. The course shows a clever trick — use Claude to write the test cases for you.

This is called synthetic data generation. You describe what kinds of inputs you want, and Claude creates them. It's like asking a classmate to quiz you instead of writing your own quiz.

📖 What the Course Teaches
Ask Claude to Generate Test Cases
generate_dataset.py Python 
import anthropic
import json

client = anthropic.Anthropic()

# ── Ask Claude to invent test cases ───────────────────────
generation_prompt = """
You are helping build a test dataset.

Generate 5 question-answer pairs for testing a customer support
AI that helps with software subscription billing.

Return ONLY a JSON array. Each item should have:
- "question": what a customer might ask
- "expected_topic": the billing topic this is about

Example format:
[
  {"question": "...", "expected_topic": "..."}
]
"""

response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=800,
    messages=[{"role": "user", "content": generation_prompt}]
)

# Claude returns JSON as text — we parse it into a Python list
raw_text = response.content[0].text
test_cases = json.loads(raw_text)

print(f"Generated {len(test_cases)} test cases!")
for case in test_cases:
    print(f"Q: {case['question']}")
    print(f"Topic: {case['expected_topic']}\n")
⬆ Even Better Approach
⬆ Why upgrade? Claude might write "easy" tests — all similar questions. A good dataset tests edge cases too.
Generate Diverse & Adversarial Test Cases

Tell Claude to intentionally include tricky, unusual, and edge-case inputs. This gives you a much more honest score.

diverse_dataset.py Python 
diverse_prompt = """
Generate 10 test cases for a billing support AI.
Make sure to include ALL of these categories:
  - 2 simple / easy questions
  - 2 angry or frustrated customer questions
  - 2 vague or unclear questions
  - 2 questions with typos or bad grammar
  - 2 completely off-topic questions (not about billing)

Return a JSON array. Each item has:
  "question": string
  "category": one of the categories above
  "expected_topic": what the AI should talk about

Only output the JSON array — no extra text.
"""

response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1200,
    messages=[{"role": "user", "content": diverse_prompt}]
)

# Strip markdown fences if Claude wraps in ```json ... ```
text = response.content[0].text.strip()
if text.startswith("```"):
    text = text.split("\n", 1)[1].rsplit("```", 1)[0]

test_cases = json.loads(text)
print(f"Generated {len(test_cases)} diverse test cases!")
💡 Analogy
It's Like Asking a Teacher to Write Your Mock Exam

A good teacher doesn't just write easy questions — they include trick questions, ambiguous questions, and hard edge cases. If you only test easy stuff, you won't know what happens when things get complicated.

⚠ Warning
Always review AI-generated test cases before using them. Claude might accidentally create test cases with wrong "expected" answers, which would make your eval scores meaningless.

Running the Eval

You have your prompt. You have your test dataset. Now it's time to run them together, collect all the answers, and measure the score. The course shows how to structure this as a clean Python script you can run any time.

📖 What the Course Teaches
The Full Eval Runner
run_eval.py Python 
import anthropic
import json

# ── Load test cases from a JSON file ──────────────────────
with open("test_cases.json") as f:
    test_cases = json.load(f)

# ── The prompt we want to evaluate ────────────────────────
SYSTEM_PROMPT = """
You are a helpful billing support assistant.
Answer customer questions about subscriptions, invoices, and payments.
Be concise and friendly.
"""

client = anthropic.Anthropic()
results = []

for i, case in enumerate(test_cases):
    print(f"Running test {i+1}/{len(test_cases)}...")

    # Send question to Claude using the system prompt we're evaluating
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=400,
        system=SYSTEM_PROMPT,           # ← the prompt under test
        messages=[{
            "role": "user",
            "content": case["question"]   # ← test input
        }]
    )

    answer = response.content[0].text

    # Quick string-match grading (we'll improve this later)
    passed = case["expected_topic"].lower() in answer.lower()

    results.append({
        "question": case["question"],
        "expected_topic": case["expected_topic"],
        "answer": answer,
        "passed": passed
    })

# ── Report results ─────────────────────────────────────────
score = sum(1 for r in results if r["passed"]) / len(results)
print(f"\n✓ Final Score: {score:.1%}")

# Save results so you can analyse them later
with open("eval_results.json", "w") as f:
    json.dump(results, f, indent=2)
⬆ Even Better Approach
⬆ Why upgrade? Running 100 tests one-by-one is slow (30+ seconds). Parallel requests make it 10x faster.
Run Tests in Parallel with asyncio
parallel_eval.py Python 
import asyncio
import anthropic

# AsyncAnthropic lets us send many requests at the same time
client = anthropic.AsyncAnthropic()

async def run_one_test(case: dict, system: str) -> dict:
    """Send one test case to Claude and return the result."""
    response = await client.messages.create(
        model="claude-opus-4-5",
        max_tokens=400,
        system=system,
        messages=[{"role": "user", "content": case["question"]}]
    )
    answer = response.content[0].text
    passed = case["expected_topic"].lower() in answer.lower()
    return {"passed": passed, "answer": answer, "case": case}

async def run_all_tests(test_cases, system_prompt):
    """Run ALL tests at the same time (in parallel)."""
    
    # Create all tasks at once — asyncio will run them together
    tasks = [run_one_test(c, system_prompt) for c in test_cases]
    
    # Wait for all tasks to finish
    results = await asyncio.gather(*tasks)
    return results

# Run it
results = asyncio.run(run_all_tests(test_cases, SYSTEM_PROMPT))
score = sum(1 for r in results if r["passed"]) / len(results)
print(f"Score: {score:.1%}")
💡 Analogy
Parallel Requests = Grading Many Papers Simultaneously

Instead of a teacher reading one paper, finishing it, then picking up the next — imagine 20 teachers each grading one paper at the same time. Parallel eval does the same thing. You finish in minutes instead of hours.

Model-Based Grading

String-matching is too simple for real prompts. If you're testing whether Claude explains a concept clearly, there's no single "expected word" to check for. That's where model-based grading comes in.

The idea: use a second Claude call (a "judge" model) to decide if the first Claude's answer was good. It's like having a teacher mark the exam instead of a spell-checker.

📖 What the Course Teaches
Using Claude as a Judge
model_grader.py Python 
def grade_with_claude(question: str, answer: str, criteria: str) -> dict:
    """
    Ask Claude to grade another Claude response.
    
    question:  What the user asked
    answer:    What Claude replied
    criteria:  What we want to check — e.g. "Is this answer friendly?"
    """
    client = anthropic.Anthropic()

    # Build the grading prompt — ask for a clear yes/no verdict
    grading_prompt = f"""
You are a grader evaluating an AI assistant's response.

ORIGINAL QUESTION:
{question}

AI'S ANSWER:
{answer}

GRADING CRITERIA:
{criteria}

Reply with EXACTLY this format:
VERDICT: PASS or FAIL
REASON: (one sentence explaining why)
"""

    response = client.messages.create(
        model="claude-opus-4-5",    # judge model
        max_tokens=150,
        messages=[{"role": "user", "content": grading_prompt}]
    )

    verdict_text = response.content[0].text
    passed = "VERDICT: PASS" in verdict_text

    return {"passed": passed, "verdict": verdict_text}

# ── Example usage ──────────────────────────────────────────
result = grade_with_claude(
    question="How do I cancel my subscription?",
    answer="Go to Settings > Billing > Cancel. You'll keep access until month end.",
    criteria="Is the answer helpful, accurate, and does it explain what happens after cancellation?"
)
print(result["verdict"])  # → VERDICT: PASS / REASON: ...
⬆ Even Better Approach
⬆ Why upgrade? A single PASS/FAIL verdict doesn't tell you how close you were to failing. A rubric score (1–5) gives you much more data.
Score on a Rubric (1–5) Instead of Pass/Fail
rubric_grader.py Python 
import re

def grade_with_rubric(question, answer):
    """
    Grade on a 1-5 scale, checking multiple dimensions:
    - Accuracy   (Is the info correct?)
    - Helpfulness (Does it solve the problem?)
    - Tone        (Is it friendly and professional?)
    """
    client = anthropic.Anthropic()

    grading_prompt = f"""
Grade this customer support response on three dimensions.
Use a scale of 1 (very bad) to 5 (excellent).

QUESTION: {question}
RESPONSE: {answer}

Rate each dimension. Reply in EXACTLY this format:
ACCURACY: [1-5]
HELPFULNESS: [1-5]
TONE: [1-5]
OVERALL_NOTES: (one sentence)
"""

    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=200,
        messages=[{"role": "user", "content": grading_prompt}]
    )
    
    text = response.content[0].text
    
    # Parse the scores out of the text with regex
    def extract_score(label):
        match = re.search(rf"{label}: (\d)", text)
        return int(match.group(1)) if match else 0

    return {
        "accuracy":    extract_score("ACCURACY"),
        "helpfulness": extract_score("HELPFULNESS"),
        "tone":        extract_score("TONE"),
        "notes":       text
    }
💡 Analogy
It's Like Getting Your Essay Marked by Another Student

Instead of just "pass or fail", a good essay rubric says: "Introduction: 8/10, Evidence: 6/10, Conclusion: 9/10." Now you know exactly what to improve. Model-based grading gives you that same depth for your AI outputs.

🎓 Pro Note
This technique is called "LLM-as-a-judge" in the research world. Studies show it closely matches human evaluation scores for most tasks, especially for open-ended responses. For high-stakes applications, you should calibrate your judge by comparing its verdicts to real human ratings.

Code-Based Grading

Sometimes you don't need an AI judge — you can write a Python function to check the answer. This is called code-based grading. It's faster, cheaper (no extra API calls), and perfectly consistent — the same code always gives the same verdict.

Use it when the right answer has a clear, checkable structure: a number, a specific word, a JSON format, or a sentence that must start a certain way.

Scenario Best Grader Why
"What is 2 + 2?" Code Answer is always exactly "4"
"Is the response in French?" Code Language detection library can check
"Is the JSON valid?" Code json.loads() will tell you instantly
"Did it explain the concept clearly?" Model "Clearly" requires understanding — only AI can judge
"Was the tone appropriate?" Model Tone is subjective — needs human-like reasoning

Grading Functions in Python
code_graders.py Python
import json import re # ── Grader 1: Check if the answer is valid JSON ──────────── def is_valid_json(answer: str) -> bool: try: json.loads(answer) # will raise error if not valid JSON return True except json.JSONDecodeError: return False # ── Grader 2: Check if answer is under 50 words ──────────── def is_concise(answer: str, max_words=50) -> bool: word_count = len(answer.split()) return word_count <= max_words # ── Grader 3: Check the answer contains a required phrase ── def contains_phrase(answer: str, phrase: str) -> bool: return phrase.lower() in answer.lower() # ── Grader 4: Check the answer starts with a bullet list ─── def starts_with_bullet(answer: str) -> bool: first_line = answer.strip().split("\n")[0] return first_line.startswith(("- ", "• ", "* ")) # ── Using them in an eval loop ───────────────────────────── test_answer = '{"name": "Alice", "age": 30}' print(is_valid_json(test_answer)) # True print(is_concise(test_answer, max_words=10)) # True (only 4 words) print(contains_phrase(test_answer, "Alice")) # True
⬆ Even Better Approach
⬆ Why upgrade? Having separate grading functions is hard to manage. A grader registry makes it easy to add, remove, and combine checks.
A Grader Registry — Combine Multiple Checks
grader_registry.py Python 
from typing import Callable, List, Dict

# ── A grader registry stores named grading functions ──────
class GraderRegistry:
    def __init__(self):
        self.graders: Dict[str, Callable] = {}

    def register(self, name: str, fn: Callable):
        """Add a grading function with a name."""
        self.graders[name] = fn

    def grade(self, answer: str, checks: List[str]) -> dict:
        """
        Run a list of named checks against an answer.
        Returns a dict of {check_name: True/False}.
        """
        scores = {}
        for check in checks:
            if check in self.graders:
                scores[check] = self.graders[check](answer)
            else:
                raise ValueError(f"Unknown grader: {check}")
        # Pass only if ALL checks pass
        scores["overall_pass"] = all(scores.values())
        return scores


# ── Set up the registry once ───────────────────────────────
registry = GraderRegistry()
registry.register("valid_json",     lambda a: is_valid_json(a))
registry.register("concise",        lambda a: is_concise(a, 50))
registry.register("has_greeting",   lambda a: contains_phrase(a, "hello"))

# ── Use it per test case ───────────────────────────────────
result = registry.grade(
    answer='{"greeting": "Hello there!"}',
    checks=["valid_json", "concise"]
)
print(result)
# → {'valid_json': True, 'concise': True, 'overall_pass': True}
💡 Analogy
Code Graders Are Like a Checklist

A pilot runs through a checklist before every flight — fuel level? ✓ Seatbelts? ✓ Landing gear? ✓ Your code graders are exactly that: a checklist of objective, measurable things an AI answer must satisfy. No subjectivity, no randomness.

✅ Tip
Best practice: Use both code-based and model-based graders together. Code checks "did it follow the format?" and model grading checks "was it a good answer?"

Build Your Own Eval System

The course exercise asks you to build a complete eval system from scratch for a simple task: a customer email classifier. Claude receives a customer email and must label it as one of: billing, technical, general, or complaint.

Your job is to write the test dataset, run the eval, and grade it. Here's a complete worked solution:

📖 Course Exercise — What's Expected
Email Classifier Eval (Course Solution)
email_classifier_eval.py Python 
import anthropic
import json

client = anthropic.Anthropic()

# ── The system prompt being evaluated ─────────────────────
CLASSIFIER_PROMPT = """
Classify the customer email into exactly one of these categories:
- billing
- technical  
- general
- complaint

Reply with ONLY the category name. Nothing else.
"""

# ── Manually written test cases ───────────────────────────
TEST_CASES = [
    {"email": "I was charged twice this month!",          "label": "billing"},
    {"email": "The app keeps crashing on my iPhone",         "label": "technical"},
    {"email": "What are your business hours?",               "label": "general"},
    {"email": "This is the worst service I have ever used!",   "label": "complaint"},
    {"email": "Invoice #12345 shows wrong amount",            "label": "billing"},
    {"email": "Can't log into my account",                   "label": "technical"},
]

results = []

for case in TEST_CASES:
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=10,           # Only need 1 word!
        system=CLASSIFIER_PROMPT,
        messages=[{"role": "user", "content": case["email"]}]
    )

    predicted = response.content[0].text.strip().lower()
    correct   = case["label"].lower()
    passed    = predicted == correct

    results.append({
        "email":     case["email"],
        "expected":  correct,
        "predicted": predicted,
        "passed":    passed
    })
    
    print(f"{'✓' if passed else '✗'} Expected: {correct:10} | Got: {predicted}")

score = sum(1 for r in results if r["passed"]) / len(results)
print(f"\nFinal Score: {score:.1%}")

⬆ Why upgrade? The basic eval has 6 hand-written tests. Combining AI-generated tests with a confusion matrix gives you much deeper insights.
Full Professional Eval with AI Dataset & Confusion Matrix
pro_email_eval.py Python 
from collections import defaultdict

def build_confusion_matrix(results: list) -> dict:
    """
    A confusion matrix shows where the model gets confused.
    
    Example:
    Actual "billing" → Predicted "complaint" 3 times
    means the model confuses angry billing emails.
    """
    matrix = defaultdict(lambda: defaultdict(int))
    
    for r in results:
        # matrix[actual][predicted] += 1
        matrix[r["expected"]][r["predicted"]] += 1
    
    return matrix

def print_confusion_matrix(matrix: dict, labels: list):
    """Print a readable confusion matrix table."""
    col_width = 12
    header = "Actual \\ Pred".ljust(col_width)
    for label in labels:
        header += label.ljust(col_width)
    print(header)
    print("-" * len(header))
    
    for actual in labels:
        row = actual.ljust(col_width)
        for predicted in labels:
            count = matrix[actual][predicted]
            # Highlight diagonal (correct predictions) with ★
            cell = (f"★{count}" if actual == predicted else str(count))
            row += cell.ljust(col_width)
        print(row)

# ── Run everything ─────────────────────────────────────────
labels = ["billing", "technical", "general", "complaint"]

# (assume results is populated from the eval run above)
matrix = build_confusion_matrix(results)
print_confusion_matrix(matrix, labels)

"""
Output looks like:
Actual \ Pred  billing     technical   general     complaint
--------------------------------------------------------------
billing        ★12         0           1           2
technical      0           ★14         0           1
general        1           0           ★10         0
complaint      3           0           0           ★8
"""

The matrix shows that "billing" emails are sometimes confused with "complaint" emails. That tells you exactly what to fix in your prompt!

💡 Analogy
A Confusion Matrix Is Like a Report Card with Details

A plain score of "70%" doesn't tell you much. But knowing "I failed Geography because I confused capital cities with largest cities" is actually useful. The confusion matrix does the same: it doesn't just say you got 70% — it says "you got billing right but kept confusing complaints with billing".

🎓 Pro Note
In production systems, evals are run automatically in a CI/CD pipeline (continuous integration). Every time a developer changes the system prompt, the eval runs automatically and blocks the change if the score drops below a set threshold (e.g. 85%). This is how serious AI teams keep their products reliable.

Everything You Learned About Prompt Evaluation
What is an Eval?
A system that tests your prompt on many inputs and measures a score — so you know if your prompt is reliable.
Eval Workflow
4 steps: write prompt → build dataset → run Claude → grade results. Repeat every time you change your prompt.
Generating Datasets
Use Claude to generate test cases including easy, hard, edge-case, and adversarial inputs. Always review them.
Running the Eval
Loop through all test cases, collect responses, and calculate a percentage score. Use async for speed.
Model-Based Grading
Use a second Claude call as a judge for open-ended responses. Can grade on rubrics (1–5) for richer feedback.
Code-Based Grading
Write Python functions to check facts: valid JSON, correct format, required keywords. Fast, cheap, and consistent.
Confusion Matrix
A table showing where your model makes mistakes. Reveals systematic errors — e.g. always confusing billing and complaints.
Best Practice
Combine code grading (format checks) + model grading (quality checks). Automate in CI/CD to catch regressions.
