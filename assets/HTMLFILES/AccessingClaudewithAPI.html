Lesson 1 — Accessing the API

Getting your API Key & Setting Up

Before Claude can work inside your app, you need two things: the right Python library installed, and a secret key that proves it's really you talking to Anthropic's servers.

1 What even is an API?

An API (Application Programming Interface) is a way for two programs to talk to each other. Think of Anthropic's servers as a restaurant kitchen — you can't walk in and cook yourself, but you can order from the menu (the API). You say what you want, the kitchen makes it, and the waiter brings it back to you.

In code terms: you write Python → it sends a message to Anthropic's servers → Claude responds → you get the answer back. Simple loop. That's the whole thing.

Real-life analogy — Zomato

Using Claude's API is like ordering on Zomato. You don't enter the kitchen. You just place an order (your message), the restaurant (Claude) cooks it up, and Zomato (the API) delivers it to you. The API is just the delivery layer between you and Claude.

2 Install the library

Open your terminal and run this one line:

Terminal 
pip install anthropic

This downloads the official Anthropic Python library — it's the "Zomato app" that handles all the messy connection details so you can just write simple Python.

3 Get your API Key

Go to console.anthropic.com, sign up for free, and navigate to API Keys → Create Key. You'll see something like:

Your API Key (example — not a real one!) 
sk-ant-api03-AbCdEfGhIjKlMnOpQrStUvWxYz1234567890-XXXXXXXXXXXXXXXXXXX

NEVER paste your API key directly in your code! If you push that code to GitHub, anyone in the world can steal it and use your account — and you'll pay the bill. Always store it in a .env file instead.

📖 What the course does

Course approach — environment variable

The course shows you to store your key in a .env file and use the python-dotenv library to load it:

.env file 
ANTHROPIC_API_KEY=sk-ant-api03-your-key-here
Python — loading the key 
from dotenv import load_dotenv
load_dotenv()  # reads .env file into environment variables

import anthropic
client = anthropic.Anthropic()
# Anthropic() automatically finds ANTHROPIC_API_KEY in env
⬆ Even better approach

Use a .gitignore too — protect yourself automatically

The course teaches the .env approach, which is great. But you should also add a .gitignore file so your .env is never accidentally pushed to GitHub. This is what professional developers do every single time.

.gitignore 
# Never push these to GitHub:
.env
.env.local
__pycache__/
*.pyc

Also — at a professional level, you'd use a secrets manager like AWS Secrets Manager or just set the environment variable directly on the server. But for learning, .env + .gitignore is perfect.

Lesson 2 — Making a Request

Your First Message to Claude

Now the fun starts. Let's send Claude a question and print the answer. This is the "Hello World" of AI programming.

📖 What the course does

The basic request — course example

The course shows the simplest possible call — one message in, one answer out:

Python — basic request (course style) 
import anthropic

client = anthropic.Anthropic()

message = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "Hello, Claude!"}
    ]
)

print(message.content[0].text)

1 What does each part mean?

Let's break apart every line — not just what it does, but why it exists:

Annotated request 
client.messages.create(

    # WHICH Claude model to use
    # claude-opus-4-6 = smartest, most expensive
    # claude-sonnet-4-6 = balanced speed/cost
    # claude-haiku-4-5 = fastest, cheapest
    model="claude-opus-4-6",

    # Max words Claude is allowed to write back
    # 1 token ≈ 0.75 words — so 1024 ≈ ~750 words
    # REQUIRED — the API will error without it
    max_tokens=1024,

    # The conversation so far, as a list
    # role: "user" = you, "assistant" = Claude
    # content: what was said
    messages=[
        {"role": "user", "content": "Hello, Claude!"}
    ]
)

2 What comes back?

The response isn't just plain text — it's a structured Python object. Here's how to navigate it:

Python — exploring the response object 
response = client.messages.create(...)

# The actual text answer:
print(response.content[0].text)

# Which model actually replied:
print(response.model)         # e.g. "claude-opus-4-6"

# Why did Claude stop? "end_turn" = finished normally
print(response.stop_reason)  # "end_turn" or "max_tokens"

# Token usage — tells you how much it cost
print(response.usage.input_tokens)   # tokens you sent
print(response.usage.output_tokens)  # tokens Claude wrote
⬆ Even better approach

Add error handling — don't let your app crash silently

The course's basic example has no error handling. In a real app, the API might fail (network error, rate limit, invalid request). Wrap it like this:

Python — with proper error handling 
import anthropic

client = anthropic.Anthropic()

try:
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": "Explain gravity simply."}]
    )
    print(response.content[0].text)

except anthropic.APIConnectionError:
    print("No internet connection. Check your Wi-Fi!")

except anthropic.AuthenticationError:
    print("Invalid API key — check your .env file!")

except anthropic.RateLimitError:
    print("Sending too many requests! Slow down.")

This way, when something goes wrong (and it will, eventually), your app tells the user what went wrong instead of just crashing with a confusing traceback.

Lesson 3 — Multi-Turn Conversations

Building a Real Back-and-Forth Chat

Claude has no memory between API calls. If you want it to "remember" what you said earlier, you have to send the entire chat history every single time. It sounds annoying — but it's actually a superpower.

Analogy — Texting without chat history

Imagine you text a friend but they delete every message the moment they read it. Each time you text them, you have to paste in the whole conversation again before asking your question. That's exactly what the API does — except your code handles the copy-pasting automatically.

📖 What the course does

Course approach — basic message history

Python — course chat loop 
conversation_history = []

while True:
    user_input = input("\nYou: ")

    conversation_history.append({
        "role": "user",
        "content": user_input
    })

    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        messages=conversation_history
    )

    reply = response.content[0].text
    print(f"Claude: {reply}")

    conversation_history.append({
        "role": "assistant",
        "content": reply
    })

1 Why does this work?

Each time you call the API, conversation_history grows by two messages (one from you, one from Claude). By the 3rd exchange, you're sending 6 messages total. Claude reads them all and can "remember" everything — because you brought the memories to it.

⬆ Even better approach

Add a memory limit — prevent runaway token costs

Here's a real problem: if your chat gets very long (say, 50 back-and-forths), you're sending thousands and thousands of tokens every time. This gets expensive fast. Professional apps keep only the last N messages:

Python — sliding window memory 
conversation_history = []
MAX_HISTORY = 10  # keep last 10 messages (5 turns)

while True:
    user_input = input("\nYou: ")
    if user_input.lower() == "quit":
        break

    conversation_history.append({
        "role": "user", "content": user_input
    })

    # 🔑 Only send the last MAX_HISTORY messages
    window = conversation_history[-MAX_HISTORY:]

    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        messages=window
    )

    reply = response.content[0].text
    print(f"Claude: {reply}")

    conversation_history.append({
        "role": "assistant", "content": reply
    })

Pro tip: Real apps like ChatGPT use a smarter version — they summarize old messages ("Earlier you asked about Python loops. We covered for, while, and list comprehensions.") and inject that summary at the top. This way, Claude never loses the big picture even if details are trimmed.

Lesson 4 — System Prompts

Giving Claude a Personality & Rules

The system prompt is a hidden instruction you give Claude before any conversation starts. It's the most powerful single thing you can do to control how Claude behaves — and most people don't use it well.

Analogy — Briefing an actor

The system prompt is like handing an actor their character brief before the play: "You play Dr. Sharma, a friendly paediatrician who explains everything in simple terms, never uses scary medical words, and always ends with a reassuring statement." From that point, every line they deliver follows those rules automatically — without you having to repeat yourself.

📖 What the course does

Course approach — basic system prompt

Python — system prompt (course style) 
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    system="You are a helpful coding assistant specializing in Python.",
    messages=[
        {"role": "user", "content": "How do I sort a list?"}
    ]
)

This is correct and it works! But a one-liner system prompt is like giving an actor one word: "Be helpful." You can do so much better.

1 The 4 Levels of System Prompts

Think of system prompts as having four increasingly powerful levels. The course teaches Level 1. Here's all four:

1
Basic Role — just a job title
Beginner 
system="You are a helpful Python tutor."
2
Role + Rules — add what to do and not do
Intermediate 
system="""You are a Python tutor for beginners.
Rules:
- Always explain concepts before showing code
- Use simple English, no jargon
- If the student seems confused, try a different analogy
- Never write more than 30 lines of code at once"""
3
Role + Rules + Format + Examples
Advanced 
system="""You are Byte, a Python tutor for 13-16 year olds.

PERSONALITY: Friendly, encouraging, uses gaming analogies.
RULES:
- Explain the concept FIRST in one sentence
- Then show the real-life analogy
- Then show the code with comments
- End with one practice challenge

FORBIDDEN: Don't say "great question!", don't use jargon,
don't write solutions to their homework.

FORMAT: Use markdown. Keep answers under 200 words unless
the student asks for more detail.

EXAMPLE OF A GOOD RESPONSE:
User: "What's a for loop?"
You: "A for loop repeats a block of code a set number
of times — like a video game telling you to collect
10 coins before the level ends..."
4
Full Production Prompt — XML structure + safety
Pro 
system="""<role>
You are Byte, an AI Python tutor built for students
aged 13-16 on the LearnCode platform.
</role>

<rules>
1. Always start with concept, then analogy, then code.
2. Use encouraging language — mistakes are normal.
3. Ask one check-in question after each explanation.
4. Never provide complete homework solutions.
5. If asked about non-Python topics, redirect kindly.
</rules>

<format>
- Use markdown headers and code blocks
- Max 250 words per response
- End with: "💡 Try this: [simple challenge]"
</format>

<safety>
- Do not discuss topics unrelated to Python or coding
- If a student mentions personal problems, show empathy
  and suggest they speak to a trusted adult
</safety>"""
⬆ Even better approach

Dynamic system prompts — personalize at runtime

The real pro move is building your system prompt dynamically using Python f-strings. This way, every user gets a system prompt tailored specifically to them:

Python — dynamic system prompt 
def build_system_prompt(student_name, skill_level, topic):
    return f"""You are Byte, a Python tutor helping {student_name}.

Student profile:
- Skill level: {skill_level}  (beginner / intermediate / advanced)
- Current topic: {topic}

Adjust your language complexity to match {skill_level}.
Always address the student as {student_name}.
Focus your examples on {topic} concepts."""

# Build a personalised system prompt for each user
system = build_system_prompt(
    student_name="Priya",
    skill_level="beginner",
    topic="for loops"
)

response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    system=system,  # personalised per user!
    messages=[...]
)

Why this matters: Every major AI product (Duolingo Max, Khan Academy's Khanmigo, Notion AI) uses dynamic system prompts to personalise the experience. It's the difference between a generic chatbot and an AI that feels like it really knows you.

The system prompt is your app's DNA — it shapes everything Claude says without the user ever seeing it.
Lesson 5 — Temperature

Controlling How Creative Claude Gets

Temperature is a number between 0 and 1 that controls how "adventurous" Claude's word choices are. It's the single most misunderstood setting in the API.

Analogy — Autocomplete on your phone keyboard

When you type "I want to eat", your phone suggests words. Low temperature = always picks the most likely word (boring but accurate). High temperature = sometimes picks surprising words (creative but occasionally weird). Claude's temperature works exactly the same way.

🧊
0.0
Ice Cold
Same answer every single time. Fully predictable. Zero randomness.
Code / Math / Facts
🌡️
0.3
Cool
Slight variation. Still reliable but with tiny vocabulary differences.
Summaries / Q&A
☀️
0.7
Warm
Noticeable variation. Responses feel natural, human, engaging.
Chatbots / Tutors
🔥
1.0
Blazing
Maximum creativity. Wild word choices. Sometimes brilliant, sometimes odd.
Stories / Poetry
📖 What the course does

Course approach — setting temperature

Python — setting temperature 
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    temperature=0.7,  # the course uses this for the exercise
    messages=[...]
)
⬆ Even better approach

Make temperature task-aware — switch it based on what you need

A real app uses different temperatures for different tasks automatically. Here's a helper that does this for you:

Python — smart temperature selector 
def get_temperature(task_type: str) -> float:
    """Pick the right temperature for the job."""
    temperature_map = {
        "code":       0.0,  # always correct, never creative
        "math":       0.0,  # one right answer
        "factcheck":  0.1,  # almost no randomness
        "summary":    0.3,  # slight variation okay
        "chat":       0.7,  # friendly, natural
        "brainstorm": 0.9,  # give me wild ideas
        "story":      1.0,  # maximum creativity
    }
    return temperature_map.get(task_type, 0.7)

# Usage examples:
task = "code"   # user wants Python help
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    temperature=get_temperature(task),  # 0.0 automatically
    messages=[...]
)

Important note: When you use Extended Thinking (Claude's step-by-step reasoning mode), temperature is fixed at 1.0 automatically — you can't change it. The thinking mode handles its own quality control internally.

Lesson 6 — Response Streaming

📖 What the course does

Course approach — basic streaming

Python — streaming (course style) 
with client.messages.stream(
    model="claude-opus-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a short poem."}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
⬆ Even better approach

Capture streamed text AND get final metadata

The course shows streaming to the terminal. In a real app (like a web server), you need to both stream text to the user AND capture it for logging, saving to a database, or adding to chat history. Here's how:

Python — stream + capture full response 
full_text = ""

with client.messages.stream(
    model="claude-opus-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Explain photosynthesis."}]
) as stream:

    # Stream to user word by word
    for text in stream.text_stream:
        print(text, end="", flush=True)
        full_text += text  # capture it too

    # After streaming ends, get full metadata
    final = stream.get_final_message()

print(f"\n\n--- Done ---")
print(f"Tokens used: {final.usage.output_tokens}")
print(f"Stop reason: {final.stop_reason}")

# Now save full_text to your database or chat history
chat_history.append({"role": "assistant", "content": full_text})

Web app tip: In a Flask or FastAPI web server, you'd yield each chunk from a route handler using Server-Sent Events (SSE). This is exactly how Claude.ai's chat interface works — it's streaming from an API route to your browser in real time.

Lesson 7 — Structured Data

Getting Clean JSON Instead of Messy Text

Usually Claude writes a nice paragraph. But what if your app needs to store Claude's answer in a database? Or pass it to another function? You need machine-readable JSON, not human-readable English.

Analogy — Order form vs conversation

Imagine asking a waiter "what's good here?" and getting a five-minute description. That's a paragraph response — great for humans. Now imagine getting a form back: {dish: "Dosa", price: 80, veg: true}. That's structured data — your app can read it instantly. Same information, completely different format.

Here's a real example showing why this matters. Ask Claude "Who are the planets?" and compare:

❌ Plain text response (hard to use in code)
The planets are Mercury, Venus,
Earth, Mars, Jupiter, Saturn,
Uranus, and Neptune. Mercury is
the smallest and closest to the
Sun, while Neptune is the
farthest away...
✅ JSON response (easy to use in code)
[
  {"name": "Mercury",
   "position": 1,
   "has_moons": false},
  {"name": "Earth",
   "position": 3,
   "has_moons": true},
  ...
]
📖 What the course does

Course approach — ask Claude to respond in JSON

Python — structured data (course style) 
import json

response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    system="Always respond in valid JSON only. No extra text.",
    messages=[{
        "role": "user",
        "content": "Give me 3 Python beginner exercises as JSON."
    }]
)

data = json.loads(response.content[0].text)
print(data)
⬆ Even better approach (2 levels)

Level 1 — Give Claude a schema to follow

Just saying "respond in JSON" works most of the time. But Claude might invent its own fields! Give it an exact schema so the output matches what your app expects:

Python — schema-guided JSON output 
schema = """{
  "exercises": [
    {
      "title": "string — name of the exercise",
      "difficulty": "easy | medium | hard",
      "topic": "string — e.g. loops, functions, lists",
      "description": "string — what the student should build",
      "starter_code": "string — Python code to start with"
    }
  ]
}"""

response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    system=f"Respond ONLY in this exact JSON format:\n{schema}",
    messages=[{
        "role": "user",
        "content": "Give me 3 beginner Python exercises."
    }]
)

import json
data = json.loads(response.content[0].text)
for ex in data["exercises"]:
    print(f"📝 {ex['title']} — {ex['difficulty']}")

Level 2 — Use Pydantic for bulletproof validation

Real production apps use Pydantic (a Python library) to validate that Claude's JSON matches the exact structure you need. If Claude accidentally writes the wrong field type, Pydantic catches it immediately:

Python — Pydantic validation (production level) 
from pydantic import BaseModel
from typing import List, Literal
import json

# Define the EXACT shape we expect
class Exercise(BaseModel):
    title: str
    difficulty: Literal["easy", "medium", "hard"]
    topic: str
    description: str
    starter_code: str

class ExerciseList(BaseModel):
    exercises: List[Exercise]

# Get response from Claude...
raw_json = response.content[0].text

# Validate it — will throw a clear error if Claude went off-script
parsed = ExerciseList.model_validate_json(raw_json)

for ex in parsed.exercises:
    print(f"✅ {ex.title} [{ex.difficulty}] — {ex.topic}")

Industry standard: The instructor Python library wraps this pattern into one elegant call: client.chat.completions.create(response_model=ExerciseList, ...). It handles the prompt injection, parsing, and retries automatically. Used by thousands of production AI apps.
